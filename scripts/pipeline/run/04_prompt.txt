# Parameter Configuration Guidelines – prompt for any EI / function module

Use this prompt to generate **only** the "Parameter Configuration Guidelines" subsection for an SAP Exception Indicator (EI) document. Provide the four input files (or their contents) under the headings at the end, then send the whole thing to the LLM.

---

## Role and task

You are documenting the **Parameter Configuration Guidelines** for an SAP Exception Indicator (EI) / function module. You will receive **four** inputs:

1. **Output structure / fields file** – structure (e.g. Excel or table) listing all output fields of the EI.
2. **Parameters (Name (Description)) file** – the **Parameters** sheet from the Available fields Excel for this EI (same as used for the Parameters Reference Table). Contains the full list of parameters for this function.
3. **ABAP code file** – the function module source code.
4. **Selected parameters file: C:\vibe code dev\ei_auto_doc\input\_Parameters.docx

**Your task:** For **every** parameter listed in the Parameters (Name (Description)) file, **in the exact order** of that file, produce a configuration guideline entry. **Group serial-number series** (e.g. KEY1–KEY10, KEY1_DS–KEY10_DS, KEY1_V–KEY10_V, or UVK01–UVK05, UVP01–UVP05, UVS01–UVS05) into one entry with a range heading and one explanation unless business meaning differs by index. Output **only** the "Parameter Configuration Guidelines" subsection. Do not output the Parameters Reference Table, Parameter Relationships, Default Values, Practical Configuration Examples, or any other section.

---

## How to write each parameter

For **each** parameter (in the order of the Parameters file):

1. **Search first in the Selected parameters file** (`_Parameters.docx`): look up the parameter by name.
   - **If an explanation exists:** Copy it and adjust only the following to match this function’s code and business logic: (a) table and field names (e.g. VBAK vs EKKO), (b) domain terms (e.g. sales order vs purchase order), (c) examples that refer to other EIs or parameters. Keep the structure and the remainder of the explanation unchanged. Do not copy verbatim if the function differs (e.g. purchase vs sales, different date fields).
   - **If no explanation exists:** Write the explanation from the function’s **ABAP code** and **business logic**. Use overall **SAP expertise** only when necessary to clarify meaning (e.g. standard tables, domains, transaction codes).

2. **Use the Output structure / fields file:** Consider all fields from this file. Tie each parameter’s semantics to the function’s **output** (e.g. which fields are filtered, aggregated, or populated). If the link is unclear from the code, use SAP expertise to explain.

3. **Parameter references:** In the main explanation and in Connection subsections, reference **only** parameters that appear in the Parameters (Name (Description)) file for this EI. Do not mention parameters that are not in that file (e.g. do not mention BACKDAYS or DATE_REF_FLD if they are not in the Parameters sheet). If the code or Selected parameters file refers to parameters not in the current function’s parameter list, do not use those names; describe behavior using only parameters that exist in the Parameters file.

4. **Structure of each parameter entry:** Every parameter has a **header** and **at least one sub-section** (the main explanation). Some parameters have **one or two additional** sub-sections. So each parameter has **1–3 sub-sections** in total.

   - **Always (1):** Main explanation – what the parameter does, how it is used in this function, which tables/fields it affects, and how it relates to the output. One or more paragraphs.
   - **Optional (2): Possible values / Options** – Include when the parameter has **specific, non-generic** values that users must choose. Apply these checks for **every** parameter: **(1) Aggregation relevance:** For each parameter, determine if it is aggregation-related by **(a) parameter name** (e.g. AGGR_PERIOD, AGGR_FIELDS, AGG_LVL, DC_IND, agg_type, aggreg_lvl), **(b) parameter description** (e.g. "Aggregation level", "Aggregation fields"), **(c) code** (how the parameter is used: grouping, aggregation logic). If **any** of (a), (b), or (c) suggests aggregation, add "Possible values / Options". Derive values from the code or Parameters file; if no fixed list exists, derive from code (e.g. single record vs aggregated) or state "Values are function-specific; see code or output structure." **(2) Boolean/flag parameters** (e.g. CHAR(1), domain XFELD or similar): **always** add "Possible values / Options" and list **all** possible values used in the code (e.g. **X**: set/active, ** ** (space): not set/inactive; include **0**/**1** if they appear in the code or domain). **(3) Other fixed-option parameters** (e.g. DATE_REF_FLD, STATUS, DURATION_UNIT): add "Possible values / Options". For **DURATION_UNIT**, when it exists in the Parameters file, you **must** use the exact wording specified in the Rules (DURATION_UNIT Options: H, M, D, F with the prescribed labels). For DATE_REF_FLD, possible values are determined from the code or table/structure-specific examples; domain fix values are not authoritative. Do **not** add Options for generic value parameters (e.g. BUKRS, company code, free-text codes) unless they are aggregation-related or boolean as above. **Default:** Keep the Options list **concise**: each bullet is the **value key** (e.g. **D**, **C**) plus a **short label** (e.g. "Debtor (customer)", "Creditor (vendor)"). Do **not** put full references (tables, filters, field names) in the Options bullets unless needed for clarity. **Exception 1 — Parameter in Selected parameters file:** If an explanation (including Options) exists in the Selected parameters file for this parameter, follow that file’s structure and level of detail for the Options subsection; the Options list may be longer if the file specifies it. **Exception 2 — Parameter not in Selected parameters file:** Keep Options short (value key + brief label). Add table/filter/field detail in the Options bullets **only when crucial** for a concise, clear explanation; otherwise put that detail in the main explanation above.
   - **Optional (3): Work together / Connection** – Include when this parameter is **designed to work together** with other parameters in most or all cases. Reference **only** parameters from the Parameters (Name (Description)) file (e.g. BACKMONTHS + COMPMONTHS, DURATION + DURATION_UNIT). Explain how they connect and how to use them together. Decide based on the function code, the Selected parameters file, and SAP expertise.

## Rules

- **Order:** Follow the **exact order** of parameters as in the Parameters (Name (Description)) file (same order as in the Parameters Reference Table prompt). Include every parameter; do not skip or merge.
- **Count:** Replace [N] in the IMPORTANT line with the actual number of parameters from the Parameters file.
- **Header format:** For each parameter use bold parameter name and the description in parentheses exactly as in the Parameters file: `**PARAM_NAME** (Description from table):`
- **Sub-sections:** Each parameter has at least the main explanation. Add "Possible values / Options" when: **(1) Aggregation-related** — for every parameter, check name, description, and code; if any suggests aggregation (e.g. AGG_LVL, AGGR_PERIOD, "Aggregation level", or code using the parameter for grouping/aggregation), add Options and derive values from code or Parameters file. **(2) Boolean/flag** (CHAR(1), XFELD-like): always add Options with all possible values (e.g. X, space, 0, 1 as in code). **(3) Other fixed-option** (e.g. DATE_REF_FLD, TIME_REF_FLD, DURATION_UNIT): add Options; for DURATION_UNIT use code-only values; for DATE_REF_FLD and TIME_REF_FLD use code-only values when the code has a literal list, otherwise use table/structure-specific examples (e.g. DATE_REF_FLD: ERDAT, BUDAT, AEDAT, GIDAT; TIME_REF_FLD: ERZET, AEZET). Add "Work together / Connection" only when the parameter clearly works together with others.
- **Fixed-option parameters (must have Options):** When the Parameters file includes **DATE_REF_FLD**, **TIME_REF_FLD**, or **DURATION_UNIT**, you **must** add a **\[PARAM_NAME] Options:** subsection for each. If the code has no literal list of values (e.g. field names are passed dynamically), still add Options with table/structure-specific examples (e.g. for DATE_REF_FLD: any date field (type D) in the table—ERDAT, BUDAT, AEDAT, GIDAT; for TIME_REF_FLD: any time field (type T)—ERZET, AEZET). **DURATION_UNIT (exact wording required):** When **DURATION_UNIT** exists in the Parameters file, you **must** add a **DURATION_UNIT Options:** subsection and list the following possible values in **exact** wording (use verbatim): H: Hours; M: Minutes; D: Days; F: Full days for specific day filtering. Use these exact heading patterns: for possible values/options, **[PARAM_NAME] Options:** (e.g. **AGGR_PERIOD Options:**, **DATE_REF_FLD Options:**, **DURATION_UNIT Options:**); for relationship between two parameters, **[PARAM_A] and [PARAM_B] Connection:** (e.g. **BACKDAYS and DATE_REF_FLD Connection:**); for a bullet list of how parameters interact, **How They Work Together:**; for combined examples, **Combined Configuration Examples:** or **Configuration Impact:**. Use the pattern that fits the content; do not invent new heading phrasings.
- **DURATION_UNIT Options (exact wording):** When the Parameters file includes **DURATION_UNIT**, you **must** add a **DURATION_UNIT Options:** subsection with the following possible values in **exact** wording (use verbatim):
  - H: Hours
  - M: Minutes
  - D: Days
  - F: Full days for specific day filtering
- **Options brevity:** For **[PARAM_NAME] Options:**, **by default** list only value keys and short explanations (e.g. `- **D**: Debtor (customer)`). Exception: DURATION_UNIT must use the exact wording above. Do not put full references (tables, filters, field names) in the Options bullets unless necessary. **If the parameter is documented in the Selected parameters file,** use that file’s structure and level of detail for Options (longer lists allowed). **If the parameter is not in the Selected parameters file,** keep Options short unless a longer Options explanation is **crucial** for clarity.
- **Source priority:** For each parameter, first use the Selected parameters file if an explanation exists; then code and business logic; then SAP expertise only when needed.
- **Tone:** Professional, business-oriented. No implementation details (no internal function names, line numbers, custom namespaces, SELECT/DELETE/WHERE/JOIN wording) in the narrative. Use standard SAP table or field names only when the guideline cannot be clear in business terms (e.g. when distinguishing two similar concepts). Prefer business wording: what the parameter is (company code, vendor, order date) and when or where it is used in the process (order header, schedule line, requisition, monitoring window).
- **Non-standard entities:** In the narrative, avoid mentioning non-standard (customer/partner) entities—e.g. names starting with `Z*`, `Y*`, or namespaces like `/SKN/*`, `/ABC4C/*`—unless absolutely necessary to explain the parameter. Prefer standard SAP object names (tables, fields, domains, transaction codes). Mention custom function modules, structures, or namespaces only when the guideline cannot be clear without them.
- **Parameter scope:** Document **only** parameters that appear in the Parameters (Name (Description)) file for this EI. Do **not** add configuration guidelines, Options, or Connection for any parameter that is not in that file (even if it appears in the ABAP code, e.g. in data_single or select_single). If the code references a parameter not in the Parameters file, do not include it in this section. In the narrative, mention only parameters from the Parameters file; do not refer to parameters that appear only in the code or in the Selected parameters file but not in the Parameters file.
- **No "output only" / "not a filter":** Do not describe any parameter as "output only," "not a filter parameter," "used for output display only," or similar. All parameters in the Parameters (Name (Description)) file can be used for filtering—at the function level (selection criteria) or at frontend/post-processing level on the result set. That distinction is irrelevant; do not state it. Describe what the parameter represents and how it is set or derived; do not classify it as filter vs. non-filter.
- **No "results limited" boilerplate:** Do not use phrases such as "Results can be limited to the given range or value," "Results are limited to the given…," "Restricts which items are included," "When supplied, only … are selected," or "Used to filter/limit/restrict the result set." Describe only **what the parameter represents** and **how it is used** in this function (e.g. company code of the PO, unit for duration). The fact that selection parameters narrow the result set is implicit; do not state it for each parameter.
- **No generic or useless sentences:** Do not include any sentence in a parameter's main explanation that is generic or filler, regardless of exact wording. A sentence is **useless** if: **(a)** it could apply to many other parameters with only the parameter name or concept swapped (e.g. "Set to focus on specific X," "Set values to narrow by X," "Parameter that restricts or populates the result set by this attribute"); **(b)** it only states that the parameter can be set, used, or applied to narrow, limit, filter, or focus the result, without adding what the parameter **is** in this function (e.g. company code of the order header, date field used for the lookback window) or **how** this function uses it in business terms (e.g. order header, schedule line, account assignment; or when the EI builds the monitoring window, when filtering by order or requisition). **Self-check:** For each parameter, ask: "If I replaced this parameter name with another, would this sentence still be true?" If yes, the sentence is generic—remove it or replace it with function- or parameter-specific content (source of the value, when/where in the process it is used, or business meaning). Each main explanation must include at least one of: where the value comes from in this function (e.g. order header, schedule line, account assignment), when or where in the process the parameter is used (e.g. when the EI reads order data, when filtering by requisition), or what it means in business terms (e.g. document currency, lookback days). Every sentence in the main explanation must add function- or parameter-specific information; remove or replace any sentence that does not.
- **Business-oriented language:** Parameter guidelines must be written for business and configuration readers. Prefer business wording over technical implementation details. Do **not** use SELECT, DELETE, WHERE, JOIN, table names (e.g. EKKO, EKPO, EKET, EBAN, EKKN), field qualifiers (e.g. K~BUKRS, P~MATNR, R_AEDAT), or other implementation terms in the main explanation unless the guideline cannot be clear without them. Instead describe **what** the parameter is (e.g. company code of the purchase order, vendor of the order, date of the requisition) and **when or where in the process** it is used in business terms (e.g. when the EI reads order header data, when filtering by account assignment, when building the monitoring window). Reserve technical references (table/field names, SELECT/DELETE) for the rare case where business meaning is ambiguous without them.
- **Currency parameters:** For currency parameters (e.g. WAERS, WAERK, WAERK_FR), describe only **business meaning**: whether the field is document/local currency, foreign currency, transaction currency, or another type. Do not include technical details such as table names, field names, or joins (e.g. do not write "from T001-WAERS" or "populated from the join to T001").
- **Date/time and lookback parameters:** For date, time, and lookback parameters (e.g. BACKDAYS, REPET_BACKDAYS, UDATE, UDATE_REPET, and similar), describe only **business meaning and effect**: what the parameter controls (e.g. "how many days to look back from today", "start of the monitoring window", "when no date range is supplied the EI uses today minus BACKDAYS"). Do **not** name internal variables, ranges, or system variables (e.g. R_DATUM, R_UDATE, SY_DATLO, SY_TIMLO, DATE_FROM, SY-DATUM). Keep the narrative business-oriented; implementation details do not add crucial value for business understanding.
- **Serial-number series:** When a series of parameters differ only by serial number, **group them into one guideline entry** with a range heading and one explanation. Serial-number series include: **(1)** KEY1–KEY10, KEY1_DS–KEY10_DS, KEY1_V–KEY10_V (key components); **(2)** same prefix + index 1–5 or 1–10, e.g. **UVK01–UVK05** (header reserves 1–5), **UVP01–UVP05** (item reserves 1–5), **UVS01–UVS05** (total reserves 1–5). Use the format **PARAM_FIRST - PARAM_LAST** (Description first – Description last): and explain the whole series once (e.g. "Key field names (1–10) for the table (TABNAME) of the change document." or "Header reserve status 1–5; each restricts which records are included and appears in the result."). **Exception:** If the same naming pattern has **genuinely different business meaning** per index, explain them individually. Do not write separate, repetitive entries when the only difference is the serial number.
- **Verification (coverage):** Before finalizing, confirm that **every parameter** from the Parameters Reference Table (section 03) is **covered** in this section: each parameter appears either as its own guideline entry or as part of a grouped entry (e.g. KEY1 through KEY10 are covered by the single entry **KEY1 - KEY10**). The number of guideline entries in this section may be **less** than the number of table rows when serial-number series are grouped; that is correct. Do not leave any parameter from the table without coverage.
- **Verification (before finalizing):** Before you declare this section done, run through this checklist: **(1)** No wording such as "output only," "output field," "not a filter," or "used for output display only"—describe what each parameter represents and how it is set or derived; do not classify as filter vs non-filter. **(2)** Every parameter from the Parameters file appears in **exact order**; only serial-number series (e.g. KEY1–KEY10, KEY1_DS–KEY10_DS, KEY1_V–KEY10_V, UVK01–UVK05, UVP01–UVP05, UVS01–UVS05) may be grouped into one entry; do not merge non-serial parameters (e.g. CUKY_NEW and CUKY_OLD are two separate entries). **(3)** Every boolean/flag parameter (CHAR(1), XFELD-like) and every other fixed-option parameter has a **\[PARAM_NAME] Options:** subsection with possible values. **(3a)** **DATE_REF_FLD**, **TIME_REF_FLD**, and **DURATION_UNIT** (when present in the Parameters file) each have an **Options** subsection. For **DURATION_UNIT**, use the exact wording prescribed in the Rules (H, M, D, F with the prescribed labels). For DATE_REF_FLD and TIME_REF_FLD use code-only values or table-specific examples (e.g. DATE_REF_FLD: ERDAT, BUDAT, AEDAT; TIME_REF_FLD: ERZET, AEZET). **(4)** Currency parameters (e.g. CUKY, WAERS) describe only **business meaning** (document/local/foreign currency); no technical details (table names, field names, joins). **(5)** Date/time and lookback parameters (e.g. BACKDAYS, REPET_BACKDAYS, UDATE): business-oriented only; no internal variable or system names (R_DATUM, R_UDATE, SY_DATLO, SY_TIMLO, DATE_FROM, SY-DATUM). **(6)** Avoid non-standard entities (Z*, /SKN/*, etc.) unless the guideline cannot be clear without them. **(7)** Main explanations are business-oriented: no SELECT, DELETE, WHERE, table names, or field qualifiers unless essential for clarity; describe what the parameter is and when/where in the process it is used in business terms.
